Данный файл содержит обязательные правила по написанию кода и структурированию проекта. Соблюдение этих правил гарантирует чистоту, масштабируемость и легкость поддержки кодовой базы.

## 🏗️ БЭМ Методология (Naming Convention)

Мы придерживаемся специфического синтаксиса **БЭМ** для именования CSS классов. Основная цель — четкая иерархия и понимание вложенности элементов.

### Формат:
`block__element-subElement`

### Правила:
1.  **Block**: Основная сущность (например, `auth`, `header`, `card`).
2.  **Element**: Составная часть блока, отделяется двумя подчеркиваниями `__` (например, `auth__content`).
3.  **Sub-element / Modifier**: Уточняющий элемент или под-часть, отделяется одним дефисом `-` (например, `auth__content-form`, `auth__title-main`).

### ⚠️ Важные ограничения:
- **Запрещены двойные дефисы**: Модификаторы и под-элементы отделяются ТОЛЬКО одним дефисом (❌ `card--blue`, ✅ `card-blue`).
- **Иерархия блоков**: Запрещено использовать дефис в названии блока, если это приводит к структуре типа `block-part__element`. Вместо этого используйте `block__part-element` (❌ `profile-feature__column`, ✅ `profile__column-left`).
- **Соблюдайте вложенность**: Элемент всегда принадлежит блоку через `__`, всё остальное — это уточнение через `-`.

### Примеры:
- ✅ `header__content` (блок + элемент)
- ✅ `header__text-item` (блок + элемент + под-элемент через дефис)
- ❌ `header-content` (неверный разделитель для элемента)
- ❌ `header-text__item` (неверный разделитель и иерархия)
- ❌ `profile__stats--active` (запрещен двойной дефис)

### 🛣️ Роутинг и Пути (Routing)
**Категорически запрещено** использовать хардкод-строки для путей (как UI, так и API) внутри компонентов.
1. **UI Пути**: Всегда через `AppRoutes` (например, `AppRoutes.LOGIN`).
2. **API Пути**: Всегда через `ApiRoutes` (например, `ApiRoutes.POSITIONS`).
3. **Версионирование**: Все API роуты должны включать версию (например, `/api/v1/...`).

## 🌐 Сетевые запросы (API)

Для работы с API в проекте используются строго определенные кастомные хуки:

1.  **`useGetQuery`**: Используется для всех GET-запросов (чтение данных).
2.  **`useMutationQuery`**: Используется для всех остальных методов (POST, PATCH, PUT, DELETE — изменение данных).

**Правила мутаций (Invalidation)**: Любое успешное изменение данных через `useMutationQuery` обязано сопровождаться инвалидацией соответствующих ключей кэша (чтобы данные `useGetQuery` обновились автоматически). **Важно**: ключи кэша/инвалидации должны храниться строго в виде `enum` или `const`-объекта (например, в `shared/config`.), использование хардкод-строк для ключей запрещено.

**Запрещено**: Использовать `axios` напрямую или создавать локальные хуки для запросов. Все запросы должны проходить через эти два инструмента для обеспечения кэширования и единообразной обработки ошибок.

1.  **Внутри компонента** (пропсы, локальные типы):
    *   Интерфейсы: `IProps` (например, `interface IProps { ... }`).
    *   Типы: `TType` (например, `type TState = 'active' | 'inactive'`).
2.  **В файле `model.ts`** (экспортируемые сущности):
    *   Интерфейсы: `I[Name]` (например, `IHeader`, `IPosition`).
    *   Типы: `T[Name]` (например, `THeaderTab`).

## 📏 Размер Файлов (File Size)

Чистота кода — наш приоритет. Мы избегаем "божественных объектов" и перегруженных модулей.

### Лимит:
**Максимальный размер ЛЮБОГО файла в проекте — 200 строк.** (Frontend, Backend, Utils, AI Services).

### Пути решения при превышении лимита:
- **Декомпозиция**: Выносите логически обособленные части в отдельные подкомпоненты или файлы.
- **Custom Hooks / Services**: Выносите сложную логику и состояние в кастомные хуки (на фронте) или отдельные сервисы (на бэке).
- **Utils**: Выносите чистые функции обработки данных в отдельные утилитарные файлы.

## 📁 Архитектура и Структура (Folder Structure)

Мы разделяем компоненты на строгие логические слои для поддержания чистоты, модульности и соблюдения однонаправленного потока данных (One-Way Data Flow).

### 🧱 Слои Архитектуры и Data Flow:
1.  **`pages/`** (Роутинг и Обертка): Страница — это исключительно точка монтирования роута.
    *   **Запрещено**: Делать здесь запросы к API (`useGetQuery`, `useMutationQuery`) или писать бизнес-логику.
    *   **Обязанность**: Максимально "тонкий" компонент. Его единственная задача — отрендерить главный компонент из слоя `widgets/` (если архитектура страницы сложная) или напрямую из слоя `features/` (если страница простая и не требует оркестрации). Получение `useParams` может происходить тут или сразу в дочернем компоненте.
2.  **`widgets/`** (Умные компоненты / Оркестраторы и Каркас): "Тяжелые" компоненты. Здесь хранятся **главные стейты** компонента и прописываются **стили главных оберток** (основной Layout/Grid страницы). Сюда выносятся сложные и крупные логические узлы (например: сайдбар, профиль целиком, сложные таблицы). 
    *   **Data Fetching**: Исключительно здесь выполняются сетевые запросы, обрабатываются состояния загрузки и ошибок.
    *   **Source of Truth**: Виджет является **главным и единственным источником пропсов**. Абсолютно все данные, стейты, методы мутаций и любые другие управляющие пропсы формируются в виджете и передаются строго **сверху вниз** — в дочерние компоненты (в слой `features/`).
    *   **Правило вызова фич**: В слое `widgets/` разрешено вызывать **только главный компонент** фичи. Запрещено вызывать внутренние подкомпоненты фичи напрямую в виджете. Вся оркестрация внутренних блоков фичи должна происходить внутри её главного компонента. Все сетевые запросы (API) и управление главными состояниями (стейты) осуществляются исключительно в виджете, после чего данные и обработчики передаются в фичу через пропсы.
3.  **`features/`** (Функциональные блоки): Принимает готовые пропсы от виджета. Главный компонент фичи не занимается роутингом. Если фича рендерится напрямую в `pages/` (минуя виджет), данные добываются ей самой через API-хуки, либо фича остается статичной.
    *   **Обязанность**: Главный компонент фичи занимается конкретикой: он распределяет пропсы и данные между своими внутренними мелкими компонентами (инпуты, карточки, таблицы) и выстраивает детальный UI внутри предоставленного каркаса.
4.  **`entities/`** (Бизнес-сущности): Содержат максимально переиспользуемые маленькие UI-компоненты (универсальные карточки, элементы списка) и **типы** данных (TypeScript интерфейсы), приходящие с сервера (`model.ts`).
5.  **`shared/`** (Общее): Абстрактный инфраструктурный слой. Не привязан к бизнес-логике конкретных страниц или фич. Обязательная внутренняя структура слоя `shared/`:
    *   `config/`: Глобальные конфигурации (константы, ключи кэша, настройки).
    *   `hooks/`: Глобальные кастомные хуки (например, `useGetQuery`, `useMutationQuery`).
    *   `lib/`: Глобальные утилиты и вспомогательные функции.
    *   `ui/`: Базовые UI-компоненты (UI Kit: кнопки, инпуты — обертки над Ant Design).
    *   `index.ts`: Единая точка входа (Public API) слоя.
    *   `model.ts`: Глобальные типы и интерфейсы.

### 🌊 Итоговый путь данных (Data Flow):
> **Pages** (Только роутинг) 👉 **Widgets** *(опционально для сложных страниц)* (Запросы к API, главные стейты, главные обертки, генерация всех пропсов) 👉 **Features** (Прием данных, конкретика UI) 👉 **Features/ui/** (Отдельные изолированные блоки UI).


### 📂 Внутренняя структура слоя Pages:
*   **Исключение из общих правил**: В слой `pages/` мы **не добавляем** файлы `index.ts`, `model.ts` или `lib.ts`.
*   Страница представляет собой минимальный файл (например, вид `PageName.tsx` или просто `ui.tsx`), который выполняет исключительно роль роута и импортирует готовые компоненты из `widgets/` или `features/`.

### 🧱 Backend (Service-Driven):
Соблюдаем строгий однонаправленный поток данных:
1.  **`routes/`**: Прием запросов, базовая валидация, вызов сервиса и ответ. Бизнес-логика здесь запрещена.
2.  **`services/`**: Основная бизнес-логика. Оркестрация данных и вызовы БД.
3.  **`db/`**: Низкоуровневая работа с базой данных (SQL запросы).
4.  **`config/`**: Константы и настройки (AI, DB, Auth).

### 📂 Внутренняя структура модуля (Entities / Features / Widgets):
**Обязательный набор файлов в каждом модуле**:
1.  **`index.ts`**: Public API. Экспортирует только необходимое.
2.  **`ui.tsx`**: Главный компонент (всегда в корне папки).
3.  **`model.ts`**: Типы, интерфейсы и логика валидации форм.
4.  **`lib.ts`**: Утилиты, константы, вспомогательные функции и колонки таблиц.
5.  **`ui/`**: Папка для вспомогательных подкомпонентов (если файл > 200 строк).

## 🔗 Экспорт и Пути (Public API & Aliases)

Для поддержания модульности и чистоты импортов:

1.  **Единая точка входа**: Обязательный файл `index.ts` в корне каждой функциональной папки. Все внешние импорты идут только через него.
2.  **Aliases**: Импорты между модулями осуществляются только через алиасы (например, `@/features/auth`) и точку входа `index.ts`. Избегайте глубоких импортов (например, `@/features/auth/ui/LoginForm`).

## 🚀 Оптимизация, Безопасность и Best Practices

1.  **Ленивая загрузка (Lazy Loading)**: Тяжелые и изначально скрытые слои UI (модалки, большие выпадающие списки, скрытые вкладки) должны подгружаться асинхронно через `React.lazy()`.
2.  **Правила мемоизации**:
    *   Применяйте `React.memo` для тяжелых дочерних компонентов.
    *   Строго запрещено передавать функции по дереву свойств вниз (в мемоизированные дочерние компоненты) без оборачивания их в `useCallback`.
3.  **Предохранители (Error Boundaries)**: Все крупные блоки (визуально обособленные `Widgets` или `Pages`) должны быть обернуты в ErrorBoundary. Падение одной таблицы или формы не должно «крашить» все приложение — должна отрисовываться локальная заглушка.
4.  **Магические строки и Хардкод**: Абсолютный запрет на использование «магических» чисел, неочевидных статусов или системных строк прямо в логике или разметке. Все бизнес-константы должны быть вынесены в `enum` или `const`-объекты в `model.ts` или `shared/config/`.

- Применяйте **vibrant colors** и **smooth transitions** для создания премиального UX.
- Код должен быть самодокументированным и аккуратным.

## 🎨 Стилизация (Тайвинд / Tailwind CSS)

- **Мандаторный Important**: Если мы используем утилитные классы Tailwind CSS, мы обязаны использовать модификатор `important` для всех классов. Символ `!` **always** ставится в конце класса (постфикс). Использование префикса (например, `!flex`) категорически запрещено.
- **Пример**: `bg-red-500!`, `text-center!`, `flex!`, `items-center!`, `m-0!`.

## 🤖 Интеграция AI (Backend & Frontend)

1. **Service-Driven**: Запрещено писать логику обращения к AI напрямую в роутах. Используйте `AiService` для оркестрации и `tools.service.ts` для реализации конкретных навыков ИИ.
2. **Environment Only**: Названия моделей (`AI_MODEL`) и URL хоста (`AI_HOST`) берутся строго из `.env`.
3. **Tool-Use Architecture**: 
   - Все технические действия ИИ (чтение БД, поиск по файлам) выносятся в `tools.service.ts`.
   - ИИ должен уметь вызывать инструменты через `tool_calls`.
   - Роут генерации обязан поддерживать цикл «Вызов инструмента -> Результат -> Финальный ответ».
4. **SSE Consistency**: Стриминг всегда возвращает `data: { content: string }` и заканчивается маркером `[DONE]`. Внутренние технические чанки (вызовы инструментов) не должны стримиться пользователю.
5. **Context & Awareness**: 
   - Всегда передавайте историю сообщений (`messages`) для сохранения контекста.
   - Обязательно передавайте `currentPath` (текущий URL), чтобы ИИ понимал, где находится пользователь.
   - Используйте `ai.knowledge.ts` для хранения базы знаний о проекте.
6. **Feedback Loop**: Каждый ответ ИИ должен иметь возможность оценки (👍/👎). Все оценки сохраняются в таблицу `ai_feedback` для последующего анализа качества ответов.
7. **UX & Aesthetics**: 
   - Используйте **Skeleton Screens** для индикации ожидания ответа.
   - Поддерживайте **Text-to-Speech** (озвучку) через Web Speech API.
   - Ответы ИИ должны рендериться через `react-markdown` с поддержкой интерактивных ссылок (Actions).
8. **Self-Initialization (Bootstrapping)**:
   - ИИ должен уметь разворачивать структуру проекта и базу данных с нуля.
   - Если файл БД (`database.sqlite`) отсутствует, метод `initDb` обязан автоматически создать его и инициализировать все необходимые таблицы (миграции).
   - При создании новых модулей ИИ обязан сразу создавать полную структуру (index, ui, model, lib).
